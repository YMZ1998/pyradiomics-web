# 放疗影像组学预测平台总体架构方案

## 方案综述

本平台拟采用 Python 全栈技术构建一个面向放疗场景的影像组学预测系统，整体架构分为前端 UI 层、后端服务层、算法处理层和数据层四个部分。前端基于 React 实现符合医疗软件操作习惯的工作站式界面，结合 Plotly 完成统计图表展示，并通过 Cornerstone 实现 CT/MR 医学影像浏览与 ROI 可视化；后端采用 FastAPI 提供高性能接口服务，并结合 Celery 与 Redis 实现异步任务调度，避免长时间计算阻塞临床操作；算法层构建标准化 Radiomics Pipeline，覆盖影像预处理、特征提取、特征筛选、模型预测与可解释性分析；数据层基于 pydicom 与 SimpleITK 处理 DICOM 影像及结构数据，并结合 SQLite 或 PostgreSQL 管理患者信息、特征结果和预测记录。系统最终形成一个集影像浏览、特征分析、AI 预测与结果解释于一体的智能辅助决策平台，可服务于放疗疗效预测、毒性风险评估及后续多中心扩展应用。

## 架构分层

### 1. 前端 UI 层

前端采用 React 构建医疗软件风格界面，整体交互设计参考 Eclipse 等临床放疗系统，以降低医护人员的学习成本和使用门槛。界面层重点集成以下能力：

- 使用 Cornerstone 进行 CT/MR 影像浏览、切片切换、窗宽窗位调整及 ROI 轮廓显示
- 使用 Plotly 展示特征统计结果、模型预测概率和解释性图表
- 构建典型医疗工作站布局，包括左侧患者树、中间影像视图和右侧分析面板
- 在 Predict 模块中划分为 `Feature`、`Predict`、`Explain` 三个子界面，分别承担特征分析、预测结果展示和模型解释任务

### 2. 后端服务层

后端基于 FastAPI 提供统一 API 服务，负责请求接收、参数校验、任务调度、状态查询以及结果返回。针对影像组学特征提取和模型推理这类耗时任务，引入 Celery 与 Redis 组成异步任务执行框架，实现任务排队、执行状态管理和结果回传，从而保证前端操作流畅性，并为后续批处理和多用户并发提供基础支撑。

### 3. 算法处理层

算法层构建标准化 Radiomics Pipeline，覆盖如下核心步骤：

- 影像预处理：包括重采样、归一化、灰度离散化等操作，以保证不同设备和中心数据的一致性
- 特征提取：基于 PyRadiomics 提取形状、强度、一阶统计、纹理等多维影像组学特征
- 特征筛选：采用低方差过滤、相关性分析、LASSO 等策略进行冗余特征剔除与关键特征筛选
- 模型预测：支持 Logistic Regression、Random Forest、XGBoost 等常见模型，并保留后续扩展深度学习模型的能力
- 可解释性分析：通过 SHAP 实现全局与个体层面的模型解释，提高预测结果的透明度和可信度

系统最终输出疗效预测概率、毒性风险预测概率及相应解释结果，以支持临床辅助决策。

### 4. 数据层

数据层以 pydicom 和 SimpleITK 为核心，负责 DICOM 影像、RT Structure 等数据的读取、转换与组织，同时使用 SQLite 或 PostgreSQL 管理患者信息、任务记录、特征数据、模型版本和预测结果。为了提高重复查询场景下的响应效率，系统设计基于“患者 + ROI + 模型”组合键的缓存机制，避免重复执行相同的特征提取和推理流程。

## 核心数据流

系统的典型业务流程如下：

1. 用户在前端选择患者、影像序列和目标 ROI，并发起预测请求。
2. 请求通过 API 提交到后端服务层，由任务队列统一调度。
3. 算法服务依次完成影像预处理、Radiomics 特征提取、特征筛选、模型推理及 SHAP 解释分析。
4. 后端将预测结果、关键特征及解释信息封装为结构化 JSON 返回前端。
5. 前端以图表、数值卡片和影像联动的方式展示最终结果，支持用户从图像、特征和预测三个维度进行综合判断。

该数据流设计能够实现“重计算后台执行、轻交互前台呈现”的解耦模式，更符合临床业务对效率与稳定性的要求。

## 可解释性设计

为增强模型的临床可理解性和可信度，平台引入 SHAP 作为核心可解释性方法，用于分析不同特征对模型输出的影响。系统支持：

- 全局特征贡献排序，展示整体样本中影响预测结果最显著的关键特征
- 个体样本解释，分析单个患者预测值由哪些特征驱动
- waterfall 图等可视化形式，用于展示特征正负贡献及累积影响过程
- 高风险与低风险病例的特征方向分析，帮助医生理解模型判别逻辑

通过上述设计，平台不仅给出预测结果，也能为结果提供可追溯、可解释的证据支持。

## 产品与界面设计

平台 UI 采用典型医疗软件布局：

- 左侧区域为患者树，用于管理患者、影像序列和 ROI
- 中间区域为影像视图，用于显示 CT/MR 图像和 ROI 轮廓
- 右侧区域为分析面板，用于展示特征统计、预测结果和解释图表

在业务流程上，系统围绕 `Feature`、`Predict`、`Explain` 三个功能子界面展开，既符合从数据到结果再到解释的认知顺序，也便于后期进行模块化扩展和权限控制。

## 工程扩展性

整体架构具备良好的工程扩展能力，可逐步支持：

- 多模型管理与模型版本控制
- 多中心数据适配与异构影像来源兼容
- 批量任务执行与任务审计追踪
- 临床级权限控制、日志记录与合规审计
- 与医院信息系统、科研数据库或其他 AI 服务的进一步集成

因此，该平台既可作为科研型原型系统快速验证，也可作为后续向医疗级智能辅助决策平台演进的基础架构。
